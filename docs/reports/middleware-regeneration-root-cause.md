## Exact Root Cause

### Trigger: AI Large-Scope Commits

Every add (`A`) commit shares a signature:

1. **Large commit touching 10–30+ unrelated files** — reports, blog components, CSS, catalog JSON, lib files, and `src/middleware.ts`
2. **Commit message is broad** — "feat: update various reports with new generation timestamps, enhance layout performance metrics, and refine blog components for improved user experience and SEO visibility"
3. **No deliberate intent to add middleware.ts** — the file appears incidentally, bundled with unrelated changes

### Why the AI Adds It

The AI (Claude Code) adds `src/middleware.ts` because:

1. **Training data bias**: Next.js documentation and thousands of open-source repos use `middleware.ts` as the standard entry point. The model's prior is that every Next.js project should have it.

2. **The file is its own honeypot**: The deleted file's docstring explicitly says:
   > "Next.js ONLY recognises middleware from a file literally named `middleware.ts`... Bug history: proxy.ts previously exported only `proxy` (named) with no `middleware.ts` shim, which left `middleware-manifest.json` empty and silently skipped all middleware work."
   
   This explanation is persuasive to an LLM reading git history. On seeing `proxy.ts` without `middleware.ts`, the model concludes that `middleware.ts` must have been accidentally deleted and restores it to "fix the bug."

3. **`proxy.ts` exports look like an incomplete shim**: `export async function proxy(...)` without the `export { proxy as middleware }` re-export pattern that Next.js tooling expects is unusual enough that the model attempts to add the missing indirection.

4. **Context window pollution**: When a large refactoring task inspects many files, `middleware.ts` gets included in the "what this codebase has" inventory via git log, which reinforces the model's belief that it belongs.

### Why Docker Doesn't Stop It

`scripts/docker-build-production.sh` already has a defensive guard:

```bash
if [[ -f src/middleware.ts || -f src/middleware.js ]]; then
  rm -f src/middleware.ts src/middleware.js
fi
```

This guard removes the file at **build time** but does not prevent it from being committed to the repository. The file exists in git for potentially hours before a build runs.

### Exact Commit Path

```
AI agent tasked with large refactor
  → scans repo context (git log, file tree)
  → observes: proxy.ts exists, middleware.ts absent
  → concludes: middleware.ts was accidentally deleted
  → adds src/middleware.ts with export { proxy as middleware, config } from "./proxy"
  → bundles into large multi-file commit
  → human or CI does not notice among 20+ other changed files
  → middleware.ts is in HEAD
  → human notices → deletes middleware.ts → commits → cycle repeats next AI session
```

## Why proxy.ts Replaces middleware.ts

Next.js 16+ recognizes `src/proxy.ts` as an alternative to `src/middleware.ts`. When **both exist simultaneously**, the build hard-fails:

```
Both middleware file ... and proxy file ... are detected
```

The correct architecture for this repo is `proxy.ts` only. `middleware.ts` is permanently banned because:
- `proxy.ts` is the single source of truth for edge routing
- Dual-registration causes a hard build failure
- The `middleware-manifest.json` is populated correctly by `proxy.ts` alone (confirmed by `proxy.ts`'s `export const config = { matcher: [...] }`)

## Protection Implemented

### 1. Contract Test — CI fails if `src/middleware.ts` exists

**File:** `src/middleware-ban.contract.test.ts`  
**Run:** `node --import tsx --test src/middleware-ban.contract.test.ts`  
**Script:** `npm run verify:no-middleware`

The test:
- Asserts `src/middleware.ts` does not exist
- Asserts `src/proxy.ts` DOES exist with correct exports
- Asserts `proxy.ts` has `export const config` with a matcher
- Documents the exact ban reason in the failure message

### 2. .gitignore entry

`src/middleware.ts` added to `.gitignore` so git refuses to stage it.

### 3. npm script

```bash
npm run verify:no-middleware
```

### 4. Playwright smoke test

**File:** `tests/e2e/smoke/proxy-registration.spec.ts`  
Verifies that Next.js middleware is actually running (edge auth headers present, x-nn-request-pathname set) confirming `proxy.ts` is wired correctly — if `middleware.ts` were absent AND `proxy.ts` weren't working, this test would catch it.

### 5. CI gate addition

`deployment-gates.yml` extended with `middleware-ban` job that runs `verify:no-middleware` on every PR and push to main.
