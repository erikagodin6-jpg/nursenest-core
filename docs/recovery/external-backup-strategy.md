# NurseNest External Backup Strategy

## Overview

This document outlines the recommended backup redundancy strategy for NurseNest.
Following the 3-2-1 backup rule: maintain at least 3 copies of data, on 2 different
types of media, with 1 copy stored off-site.

## Backup Tiers

### Tier 1: Local Backups (Primary)
Generated by `npm run backup:full` and stored in the `/backups/` directory.

- **Frequency**: Daily snapshots, weekly full backups
- **Retention**: Keep last 7 daily, last 4 weekly
- **Content**: Code, DB schema, migrations, content manifests, asset indexes
- **Scripts**:
  - `scripts/backup/run-daily-snapshot.ts` - Code + DB + manifests
  - `scripts/backup/run-weekly-full.ts` - Complete backup with archives

### Tier 2: Cloud Storage (Secondary)
Upload backup archives to a cloud storage provider.

#### Google Drive
1. Install Google Drive desktop client or use rclone
2. Create a `NurseNest-Backups` folder
3. Copy backup archives after each weekly backup:
   ```bash
   cp backups/nursenest-backup-*.zip ~/GoogleDrive/NurseNest-Backups/
   ```

#### Dropbox
1. Create a `NurseNest-Backups` folder in Dropbox
2. Use Dropbox CLI or desktop sync:
   ```bash
   cp backups/nursenest-backup-*.zip ~/Dropbox/NurseNest-Backups/
   ```

#### Amazon S3 (Recommended for Production)
1. Create an S3 bucket: `nursenest-backups`
2. Configure lifecycle rules to move to Glacier after 30 days
3. Upload using AWS CLI:
   ```bash
   aws s3 cp backups/nursenest-backup-*.zip s3://nursenest-backups/weekly/
   ```
4. Enable versioning on the bucket for extra protection

### Tier 3: Version Control (Tertiary)
Maintain a private GitHub repository as a versioned backup.

1. Create a private repo: `nursenest-backup-archive`
2. After each backup, commit key non-binary files:
   ```bash
   git add backups/manifests/ backups/db/schema.ts backups/db/table-row-counts.json
   git commit -m "Backup: $(date +%Y-%m-%d)"
   git push origin main
   ```
3. Do NOT commit large zip archives to git (use .gitignore)
4. Do NOT commit any files containing secrets or credentials

## What to Back Up

### Always Back Up
- Source code (client/, server/, shared/, scripts/)
- Database schema and migrations
- Environment variable template (.env.example)
- Content manifests and indexes
- Configuration files (package.json, tsconfig.json, etc.)

### Back Up Periodically
- Database content exports (question banks, lessons, flashcards)
- Asset indexes
- Translation files
- Blog and article slugs

### Never Back Up
- node_modules/ (regenerated via npm install)
- dist/ (regenerated via npm run build)
- .env files with real secrets
- Session data
- Temporary files

## Recovery Priority

In a disaster scenario, restore in this order:

1. **Code**: Extract backup archive or clone repository
2. **Database**: Apply migrations, import content data
3. **Configuration**: Set up environment variables
4. **Assets**: Restore uploaded images and files from object storage backup
5. **Verification**: Run `npm run restore:validate`

## Testing Your Backups

Run these commands monthly to verify backup integrity:

```bash
npm run restore:validate
npm run restore:dry-run
```

These will confirm all critical files, configs, and database components are present
and a restore would succeed.

## Automation (Future)

The following scripts are prepared for automated scheduling:
- `scripts/backup/run-daily-snapshot.ts` - Daily at 2:00 AM
- `scripts/backup/run-weekly-full.ts` - Sunday at 3:00 AM
- `scripts/backup/run-predeploy-snapshot.ts` - Before each deployment

To automate with cron (when running on a dedicated server):
```cron
0 2 * * * cd /app && npx tsx scripts/backup/run-daily-snapshot.ts
0 3 * * 0 cd /app && npx tsx scripts/backup/run-weekly-full.ts
```
